# 주식 거래소 마이크로서비스 아키텍처 요건

## 핵심 도메인 서비스

**거래 처리 (Trading Engine)**

- 주문 접수/검증/매칭 엔진 (Order Book)
- 체결 처리 및 청산 서비스
- 지정가/시장가/조건부 주문 타입 처리

**계좌 & 자산 관리**

- 사용자 계좌 서비스 (잔고, 포트폴리오)
- 증거금 관리 및 리스크 한도 서비스
- 출금/입금 처리 서비스

**시세 & 데이터**

- 실시간 시세 피드 서비스 (WebSocket 기반)
- 호가창(Order Book) 집계 서비스
- 시계열 OHLCV 데이터 서비스

**정산 & 청산**

- T+2 정산 처리 서비스
- 배당/권리락 처리
- 세금 계산 서비스



## 인프라 & 크로스커팅 요건

**메시지 브로커 (핵심)**

- Kafka: 주문 이벤트 스트림, 체결 이벤트 브로드캐스트
- 이벤트 소싱 패턴으로 모든 거래 이력 보장
- Dead Letter Queue로 실패 이벤트 처리

**API Gateway & 인증**

- JWT + OAuth2 기반 인증/인가
- Rate Limiting (API 남용 방지)
- Circuit Breaker (Resilience4j 등)

**데이터 저장소 전략**

| 용도             | 기술                   |
| ---------------- | ---------------------- |
| 주문/체결 이벤트 | Kafka (이벤트 소싱)    |
| 계좌/잔고        | PostgreSQL (ACID 필수) |
| 시세 시계열      | TimescaleDB / InfluxDB |
| 세션/캐시        | Redis                  |
| 검색/로그        | Elasticsearch          |



## 비기능적 요건 (NFR)

**성능**

- 주문 매칭 레이턴시: **< 1ms** (거래소 핵심 KPI)
- 초당 처리량 TPS: 수만 건 수준 목표
- 시세 피드 지연: < 10ms

**신뢰성**

- 멱등성 보장 (중복 주문 방지)
- 분산 트랜잭션 → Saga 패턴 적용
- 서킷 브레이커로 장애 전파 차단

**보안**

- 모든 금융 데이터 암호화 (at-rest, in-transit)
- 감사 로그 (Audit Log) 전 서비스 필수
- FIDO2/MFA 인증

**규제 준수**

- 주문 이력 5~10년 보관 (금융법 의무)
- 이상 거래 탐지 (FDS) 서비스 별도 구성
- 거래 보고 서비스 (감독기관 리포팅)



## 서비스 간 통신 패턴

```
사용자 → API Gateway
         ├── 동기(REST/gRPC): 주문 접수, 잔고 조회
         └── 비동기(Kafka):   체결 이벤트, 알림
                   ↓
         Order Service → Matching Engine → Settlement Service
                                ↓
                         Account Service (잔고 업데이트)
```
