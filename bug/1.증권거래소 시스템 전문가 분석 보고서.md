### 1-2. 주문 생명주기(Order Lifecycle) 적합성

**FIX 5.0 표준 기준 vs 현재 구현**:

```
[FIX 5.0 표준]
New → PendingNew → New → PartiallyFilled → Filled
                        → Cancelled
                        → Rejected
                        → Expired (IOC/FOK/GTD)

[현재 구현]
PENDING → (매칭 엔진) → 체결 이벤트 (DB 상태 자동 미갱신)
```

**차이점**:
- `PARTIALLY_FILLED` 상태가 도메인 엔티티에 없음 → 부분 체결 상태 추적 불가
- 체결 후 order-service DB 상태가 자동 갱신되지 않음 (trading-engine → order-service 역방향 이벤트 미구현)
- `EXPIRED` 상태 없음 (IOC/FOK/GTD 주문 지원 불가)

### 1-3. 호가창 Price-Time Priority 평가

```java
// bids: ConcurrentSkipListMap(reverseOrder) → 높은 가격 우선 ✅
// asks: ConcurrentSkipListMap(naturalOrder)  → 낮은 가격 우선 ✅
// 동일 가격: TreeSet<OrderBookEntry>(sequence 오름차순) → FIFO ✅
```

Price-Time Priority 원칙은 **올바르게** 구현되었습니다. 단, 매칭과 스냅샷 간 동시성 Lock 범위 불일치 문제가 존재합니다(MAJOR-02 참조).

## 2. 수정이 필요한 사항

---

### 🔴 CRITICAL-01. 증거금 동결과 주문 저장의 원자성 결여

- **위험도**: Critical — 자금 손실 유발 가능
- **위치**: `OrderCommandService.java` L66-86
- **문제점**:

```java
// 1. 증거금 동결 (account-service REST 호출) ← 외부 I/O
boolean frozen = accountServiceClient.freezeBalance(...); // 성공

// 2. 주문 저장 + Outbox 저장 (@Transactional)
Order savedOrder = orderCommandPort.save(order); // ← DB 실패 가능!
outboxService.save(...);
```

증거금 동결 성공 후 DB 트랜잭션이 실패하면 **잔고는 동결되었으나 주문이 없는 상태**가 됩니다. 반대로 DB 성공 후 네트워크 이슈가 발생하면 동결 여부를 알 수 없습니다. 전형적인 **TOCTOU(Time-of-Check Time-of-Use)** 결함입니다.

- **권고 수정 방향**:
  - **Saga Pattern (Choreography)** 도입: `ORDER_CREATED` 이벤트 발행 → account-service가 소비하여 동결 → `BALANCE_FROZEN` 이벤트로 주문 상태 확정
  - 보상 트랜잭션(Compensating Transaction) 필수 구현
- **참고**: Saga Pattern (Chris Richardson), FIX 5.0 `PendingNew` 상태 분리 원칙

### 🔴 CRITICAL-02. StopOrderManager 인메모리 상태 — 재시작 시 전량 소실

- **위험도**: Critical — 고객 손절 주문 미작동 → 예상치 못한 손실
- **위치**: `StopOrderManager.java` L32-33

```java
private final ConcurrentHashMap<String, CopyOnWriteArrayList<StopOrderEntry>> pendingStops
        = new ConcurrentHashMap<>(); // JVM 메모리에만 존재!
```

trading-engine 재시작 시 **등록된 모든 STOP 주문이 소실**됩니다. 클러스터 환경(다중 인스턴스)에서는 각 인스턴스가 서로 다른 STOP 목록을 가집니다.

- **권고 수정 방향**:
  - STOP 주문을 **Redis Sorted Set** 또는 DB 테이블에 영속화
  - 서비스 시작 시 `@PostConstruct`로 DB에서 PENDING STOP 주문 복구
  - order-service DB에도 `STOP_QUEUED` 상태로 기록
- **참고**: KRX 조건부 주문 운영 규정, CME Globex Stop Order Handling

### 🔴 CRITICAL-03. 부분 체결(Partial Fill) 시 정산 누락

- **위험도**: Critical — 자금 정산 누락
- **위치**: `SettlementService.java` L49, L73

```java
// 부분 체결이 2번 발생하는 경우:
if (!settlementRepo.existsByOrderIdAndSide(buyOrderId, "BUY")) {
    // 1번째 체결만 저장됨
    // 2번째 체결 수량은 이 조건에 걸려 정산 레코드 미생성!
}
```

부분 체결이 여러 번 발생할 때 **첫 번째 체결만 정산**되고 나머지 체결 수량은 정산 레코드가 생성되지 않습니다.

- **권고 수정 방향**:
  - `ExecutionResult`에 고유 `executionId`(UUID) 부여
  - 중복 체크를 `(orderId, executionId)` 복합 조건으로 변경
  - 체결 건별로 독립 정산 레코드 생성
- **참고**: ISO 20022 `sttlm` 메시지, KRX 청산·결제 업무규정 제7조

### 🟠 MAJOR-02. OrderBook 동시성 — Lock 범위 불일치

- **위험도**: Major — 데이터 경쟁 조건, 호가창 불일치
- **위치**: `OrderBook.java` L41-48 / `MatchingEngine.java` L36

`MatchingEngine`은 `synchronized(orderBook)`으로 보호하지만, `cancelOrder()`, `OrderBookSnapshotService.saveSnapshot()`에서 `getBids()/getAsks()`를 직접 노출하여 이 Lock 밖에서 수행됩니다.

- **권고**: `OrderBook` 내부에 `ReentrantReadWriteLock` 적용 (read: snapshot, write: add/match/cancel). 고성능 요건 시 LMAX Disruptor 패턴의 **단일 스레드 매칭엔진** + Lock-Free 구조 채택

### 3-3. 성능 목표(Latency < 1ms, TPS > 100,000) 달성 로드맵

**현재 구조의 병목**:
```
주문 접수 → HTTP → DB 저장 → Outbox Polling(5초) → Kafka → 매칭
```
Outbox 폴링 지연만으로 최소 5초의 주문-매칭 지연이 발생 → **1ms 목표 달성 불가**

**단계별 개선 로드맵**:

| 단계        | 개선 항목                                   | 예상 효과                 |
| ----------- | ------------------------------------------- | ------------------------- |
| **Phase 1** | Outbox 폴링 주기 100ms 이하 단축            | 지연 5s → 0.1s            |
| **Phase 1** | Kafka 파티셔닝(symbol별) 병렬화             | TPS 3× 향상               |
| **Phase 2** | LMAX Disruptor 기반 단일 스레드 매칭 엔진   | 매칭 latency < 1μs        |
| **Phase 2** | Kafka 대체: Aeron(UDP) 또는 Chronicle Queue | 네트워크 latency 90% 감소 |
| **Phase 3** | Kafka Transactions + Idempotent Producer    | Exactly-Once 보장         |
| **Phase 3** | GraalVM Native Image 또는 ZGC 튜닝          | Stop-the-World 제거       |
| **Phase 4** | Kernel Bypass (DPDK/RDMA)                   | < 100μs latency           |

> **참고**: KRX EXTURE+ 시스템은 C++ 기반 단일 프로세스 매칭 엔진 + 공유 메모리 IPC 구조입니다. 현재의 Spring Boot + Kafka 구조는 프로토타입에 적합하나, 초저지연 요건 달성을 위해서는 **매칭 엔진 코어를 별도 프로세스로 분리**해야 합니다.
