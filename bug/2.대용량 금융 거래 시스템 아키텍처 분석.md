## 1. 시스템 레이어 분리 적합성

### 레이어별 완성도

| 표준 레이어          | 현재 구현체                                        | 완성도 |
| :------------------- | :------------------------------------------------- | :----- |
| **Gateway**          | `api-gateway` (Spring Cloud GW + JWT + Rate Limit) | ⚠️ 60%  |
| **Order Management** | `order-service` (Outbox + CQRS)                    | ✅ 80%  |
| **Matching Engine**  | `trading-engine` (Price-Time Priority)             | ✅ 75%  |
| **Risk Check**       | `account-service` (잔고 동결만)                    | ❌ 30%  |
| **Clearing**         | `settlement-service` (T+2 정산)                    | ✅ 70%  |

### 주요 문제점

**① Gateway — FIX 프로토콜 수신 부재** 브로커사 연동에 필수적인 FIX 5.0 TCP 수신 엔드포인트가 없습니다. Rate Limiter도 IP 기준만 적용되어 동일 사용자 다중 IP 우회가 가능합니다.

**② Risk Check 레이어 — 가격 제한폭 등 Pre-Trade Risk 전무**

| Risk Check 항목        | 현재 구현           |
| :--------------------- | :------------------ |
| 잔고 충분 여부         | ✅                   |
| 가격 제한폭 (±30%)     | ❌ **KRX 필수**      |
| 포지션 한도            | ❌                   |
| 서킷브레이커 발동 확인 | ❌                   |
| 자기자본 대비 한도     | ❌ **MiFID II 필수** |

**③ 순환 의존 — `@Lazy`로 회피 중**

```
java





// StopOrderManager → ExecutionService (순환 의존)

@Autowired

public StopOrderManager(@Lazy ExecutionService executionService) { ... }
```

`TriggerEvent`를 발행하고 `ExecutionService`가 구독하는 이벤트 기반 구조로 전환이 필요합니다.

## 2. 데이터 일관성 및 이벤트 처리

### 처리 보장 수준

| 구간                        | 현재 수준          | 문제                    |
| :-------------------------- | :----------------- | :---------------------- |
| order-service → DB          | ✅ Exactly-Once     | -                       |
| DB → Kafka (Outbox)         | ✅ At-Least-Once    | 재처리 시 중복 이벤트   |
| Kafka → trading-engine      | ❌ **At-Most-Once** | 예외 삼킴 → 메시지 소실 |
| trading-engine → settlement | ⚠️ At-Least-Once    | 중복 체결 이벤트 가능   |

### 핵심 문제

**① Kafka 단일 브로커 — SPOF**

```
yaml





KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1  # ❌ 브로커 1개 장애 시 전체 중단
```

→ **3-Broker Cluster + `min.insync.replicas: 2`** 구성 필수

**② Kafka Exactly-Once 미설정**

```
yaml





# 현재 기본값 (추정): acks=1, enable.idempotence=false

# 권고:

spring.kafka.producer:

  acks: all

  enable-idempotence: true

  transaction-id-prefix: order-tx-
```

**③ Consumer가 예외를 삼켜 메시지 소실**

```
java





} catch (Exception e) {

    log.error(...);

    // offset 자동 commit → 체결 이벤트 영구 소실 ❌

}
```

→ `DeadLetterPublishingRecoverer` + 수동 Offset Commit 적용 필요

**④ OrderBook Replay 불가** — trading-engine 재시작 시 호가창 초기화, 복구 전략 없음

## 3. 데이터베이스 설계

### 핵심 누락 — `Trade(체결)` 테이블이 존재하지 않음

체결 결과가 `ExecutionResult` DTO로만 처리되며 **DB에 영속화되지 않습니다**. 체결 데이터는 감리·분쟁 해결·규제 보고의 핵심 데이터입니다.

```
sql





-- 필수 추가 테이블

CREATE TABLE trades (

    trade_id     CHAR(36)       NOT NULL,

    exec_id      VARCHAR(64)    NOT NULL UNIQUE,  -- 거래소 발급 체결 ID

    buy_order_id CHAR(36)       NOT NULL,

    sell_order_id CHAR(36)      NOT NULL,

    symbol       VARCHAR(20)    NOT NULL,

    exec_price   DECIMAL(20,8)  NOT NULL,

    exec_qty     DECIMAL(20,8)  NOT NULL,

    exec_time    DATETIME(6)    NOT NULL,          -- 나노초 정밀도

    seq_no       BIGINT         NOT NULL           -- 단조 증가 체결 순번

);
```

### OHLCV — Redis 단독 저장의 한계

| 문제           | 상세                          |
| :------------- | :---------------------------- |
| 휘발성         | Redis 장애 시 OHLCV 소실      |
| 용량 한계      | 최대 500개 → 장기 차트 불가   |
| Tick Data 없음 | OHLCV 재계산 기반 데이터 없음 |

→ **TimescaleDB Hypertable + Continuous Aggregate** 도입 권고

```
sql





SELECT create_hypertable('tick_data', 'exec_time',

    chunk_time_interval => INTERVAL '1 day');



CREATE MATERIALIZED VIEW ohlcv_1m WITH (timescaledb.continuous) AS

SELECT time_bucket('1 minute', exec_time) AS bucket, symbol,

       FIRST(price, exec_time) AS open, MAX(price) AS high,

       MIN(price) AS low, LAST(price, exec_time) AS close ...
```

### 누락 인덱스

```
sql





-- 매칭 엔진 복구용 (필수)

INDEX idx_symbol_status_price (symbol, ord_status, side, price)



-- 회원 포털 조회 (필수)

INDEX idx_member_recent (member_id, transact_time DESC)
```

## 4. 누락된 아키텍처 구성요소

### ① Circuit Breaker — 설정 미흡

`FallbackController`는 있으나 서비스 간 Circuit Breaker 미설정. `RestTemplate` 타임아웃도 없어 account-service 장애 시 전체 스레드 풀 고갈 위험.

### ② Backpressure — 완전 부재

장 시작(9:00~9:05) 주문 집중 시 **모든 요청이 DB까지 도달**. Kafka `max-poll-records: 50` 제한 + Gateway Token Bucket Rate Limit 적용 필요.

### ③ DR(다재해 복구) — 완전 부재

```
현재: 단일 Docker Compose, 단일 AZ

권고: Active-Passive 이중화

      MySQL Binlog Replication → DR DC

      Kafka MirrorMaker2       → DR DC

      Redis Sentinel           → DR DC

      DNS Failover             → 자동 절체
```

### ④ Market Data L2/L3/Tick — 미구현

| 레벨                  | 현재 구현                |
| :-------------------- | :----------------------- |
| L1 (Best Bid/Ask)     | ⚠️ TickerDto (부분)       |
| L2 (상위 10단계 호가) | ⚠️ REST만, WebSocket 없음 |
| L3 (개별 주문 호가창) | ❌ 없음                   |
| Tick (체결 스트림)    | ❌ 없음                   |

## 개선 우선순위 로드맵

| 단계        | 기간           | 핵심 작업                                                    |
| :---------- | :------------- | :----------------------------------------------------------- |
| **Phase 1** | 즉시 (1~2주)   | 3-Broker Kafka, Consumer DLQ, CircuitBreaker, StopOrder DB 영속화 |
| **Phase 2** | 단기 (1~2개월) | Trade 테이블, TimescaleDB, Pre-Trade Risk Gate 서비스, Exactly-Once 설정 |
| **Phase 3** | 중기 (3~6개월) | L2/L3 Market Data, DR 이중화, Audit Trail, Event Store 분리  |
| **Phase 4** | 장기 (6개월~)  | FIX 5.0 Gateway, LMAX Disruptor 매칭 엔진, Kernel-bypass, MiFID II 보고 |

> **즉시 해결해야 할 아키텍처 위험 요소**: Kafka 단일 브로커(SPOF), Trade 테이블 미존재(체결 데이터 영구 소실), DR 구성 부재입니다.
